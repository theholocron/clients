# CLAUDE.md

Conventions for working on the `theholocron/clients` monorepo.

## Where code lives (org-wide rule)

Three repos, one rule per concern:

- **Shareable tool config (ESLint, Prettier, TSConfig, Vitest, …)** → `theholocron/configs`. If you find yourself copy-pasting a tool config across repos, it belongs there as a `@theholocron/*-config` package.
- **HTTP clients and API wrappers** → `theholocron/clients` (this repo). REST clients for third-party services (Jira, Zendesk, Google, …) and shared HTTP primitives live here.
- **Anything that can be automated** → `theholocron/holocron`. Infrastructure commands (`setup`, `upgrade`, `doctor`, `secrets sync`), CI orchestration, and repo lifecycle automation belong in the Holocron CLI.
- **`holocron.config` format** — use `holocron.config.ts` with `defineConfig` in any repo that has a `package.json` (the CLI must be resolvable at runtime). Use `holocron.config.json` in content-only repos with no Node.js infrastructure (e.g., `.github`, `.github-private`).

## Architecture

- **pnpm workspace monorepo** with Turborepo for task orchestration.
- Each package under `packages/` is an independently published npm package.
- All packages compile TypeScript source (`src/`) to `dist/` via `tsdown`.
- `@theholocron/http-client` is the canonical home for shared HTTP
  primitives (`createRestClient`, `createResolveToken`, `ProviderApiError`).
  All other clients depend on it.

## Code patterns

- **Package manager: pnpm only.** Never use `npm` or `yarn`. Run workspace-wide tasks through Turbo (`pnpm test`, `pnpm build`, etc.); run single-package tasks with `pnpm --filter <name> <script>`.
- **No `any` in TypeScript.** Use `unknown` for values of genuinely unknown shape and narrow with type guards. Use generics instead of `any` in function signatures. `as never` and `as unknown as T` are acceptable for internal casts where the type system can't follow; `any` is not.
- **ESLint override:** `n/no-unpublished-import` is disabled in every
  package's `eslint.config.js`. This is a known false positive for the
  TypeScript `src/ → dist/` build model — `files[]` in `package.json`
  lists `dist/`, so every relative `src/` import is flagged. Keep the
  rule off at project level; do not push it to the org config.
- **`.alexrc.json` is generated by `holocron setup`** — do not edit it manually. To update the alex prose-linter allow-list, edit `ALEX_CONFIG` in `theholocron/holocron packages/cli/src/commands/setup.ts` and re-run `holocron setup` here.
- **Running `holocron setup`**: `@theholocron/cli` and `@theholocron/holocron-plugin-github` are installed under `catalogs.holocron` in `pnpm-workspace.yaml` so setup is runnable locally. Use `GITHUB_TOKEN=$(gh auth token) pnpm exec holocron setup`. Workflow files, `.alexrc.json`, `.github/dependabot.yml`, and repo-policy changes (branch protection, security settings) are all applied in one pass. Bump both catalog entries together when upgrading the CLI.
- **`engines.node`** must be set in every `package.json`. To update the
  Node.js version across the whole repo run `holocron upgrade node <to>`
  from any repo that has the CLI installed — it scans and patches
  `engines.node`, `@types/node`, `.nvmrc`, `.node-version`, and
  `.github/workflows/*.yml` automatically.

## Workflow

- **Always open a PR — never push directly to the default branch.** Even for small fixes: create a branch, push it, open a PR. This lets CI run, keeps history reviewable, and respects branch protection. The only exception is bootstrapping a brand-new repo before protection is set up.
- **Conventional Commits.** `feat:` / `fix:` / `chore(deps):` / `docs:` /
  `ci:` / `test:` — semantic-release uses these for the changelog.
- **Always `git commit -s`** (DCO). `Signed-off-by:` trailer required.
- **No Claude attribution** in commits, PRs, or docs.
- **Definition of done: code + tests + docs + green checks.**

## Quality

- `pnpm test` — vitest across all packages via Turbo.
- `pnpm typecheck` — `tsc --noEmit` in each package.
- `pnpm lint` — ESLint via Turbo.
- Test helpers live in `src/__tests__/helpers.ts` per package (e.g.,
  `stubFetch` for injecting fetch responses without global stubs).
